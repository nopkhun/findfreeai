# Dashboard Operations Map

เอกสารนี้สรุปแผนที่การแก้ปัญหาเชิงปฏิบัติการของแดชบอร์ด โดยทุกแผงดึงข้อมูลจาก proxy แบบสด (live) เพื่อใช้ตรวจสอบก่อนปล่อยระบบ

## แผง: สถานะผู้ให้บริการ

- **Endpoint ต้นทาง:** `GET /v1/providers`
- **สิ่งที่แสดง:**
  - สถานะ `UP`/`DOWN`
  - คูลดาวน์ของ provider
  - สถิติสำเร็จ/ล้มเหลว/ค่าเฉลี่ย latency
- **สัญญาณเตือนหลัก:** badge มีคำว่า `DOWN`
- **แนวทางแก้ไข:**
  1. เติม API key ของ provider ที่ DOWN
  2. ตรวจคูลดาวน์จาก `/v1/providers` และ `/v1/stats`
  3. รีโหลด provider ผ่าน `GET /v1/reload`

## แผง: สถานะคีย์และคอนฟิก

- **Endpoint ต้นทาง:** `GET /v1/config`, `GET /v1/stats`
- **สิ่งที่แสดง:**
  - โหมดการทำงาน (`auto/manual/round-robin`)
  - ค่า retry/timeout แบบปกปิดข้อมูลสำคัญ
  - จำนวน provider ที่ยังไม่มีคีย์
- **สัญญาณเตือนหลัก:** คำเตือนมีข้อความ `ยังไม่ได้ตั้งค่า`
- **แนวทางแก้ไข:**
  1. เติม key ที่ขาดใน `api_keys.json` หรือ env
  2. ทดสอบ key จากแดชบอร์ด
  3. ยืนยันผลอีกครั้งผ่าน `/v1/config` และ `/v1/stats`

## แผง: คำแนะนำการแก้ไข

- **Endpoint ต้นทาง:** ประมวลผลจาก `GET /v1/providers`, `GET /v1/config`, `GET /v1/stats`
- **สิ่งที่แสดง:** รายการ remediation ตามลำดับความสำคัญ (สูง/กลาง/ปกติ)
- **แนวทางแก้ไข:**
  - ถ้า provider DOWN หลายราย: เติมคีย์และทดสอบก่อน
  - ถ้าติดคูลดาวน์: รอ cooldown หรือปรับโหมดเป็น manual ชั่วคราว
  - ถ้าไม่มีความผิดปกติ: คงค่าเดิมและเฝ้าระวังผ่าน proxy log

## จุดตรวจสอบเร็ว (Release Gate)

ก่อนปล่อยระบบให้ตรวจตามลำดับ:
1. แผง `สถานะผู้ให้บริการ` ไม่มี DOWN ที่สำคัญ
2. แผง `สถานะคีย์และคอนฟิก` ไม่มีคำเตือน `ยังไม่ได้ตั้งค่า` สำหรับ provider ที่ต้องใช้งานจริง
3. แผง `คำแนะนำการแก้ไข` เหลือเฉพาะระดับ `ปกติ`
